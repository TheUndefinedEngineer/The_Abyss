# Serial Peripheral Interface

> The foundation of the entire xSPI family (QSPI, OSPI etc.).
> Understanding SPI well makes QSPI and OSPI much easier to reason about.

---

## What is SPI?

**Serial Peripheral Interface** — a synchronous, full-duplex serial communication protocol originally developed by Motorola in the early 1980's.

- **Synchronous** → master generates a clock; all data is aligned to it
- **Full-duplex** → data flows in both directions simultaneously
- **Master-Slave** → one master drives the clock and chip select; slaves respond
- **No formal standard** → de-facto, so vendors have minor variations

SPI uses Bus topology of communication - [[Topologies Of Communication]]

---

## The 4 Signals

```
SCLK  (Serial Clock)       — generated by master, shared by all slaves
MOSI  (Master Out Slave In) — data from master → slave
MISO  (Master In Slave Out) — data from slave → master
CS/SS (Chip Select)        — active LOW, one per slave
```

> SCLK, MOSI, MISO are shared on the bus. Each slave gets its own CS line — this is how the master selects who it's talking to.

Modern terminology (per JTAG/open-hardware community):
- MOSI → **COPI** (Controller/Master Out Peripheral In)
- MISO → **CIPO** (Controller/Master In Peripheral Out)

Both names mean the same thing — both can be seen in datasheets.

---

## How a Transaction Works

```
1. Master pulls CS LOW  → slave is selected
2. Master drives SCLK
3. Data shifts out on MOSI (master → slave) bit by bit
4. Data shifts in on MISO (slave → master) simultaneously
5. Master pulls CS HIGH → transaction ends
```

- One bit is transferred per clock cycle.
- For 1 byte = 8 clock cycles.
- MSB (Most Significant Bit) is sent first by default — but some devices do LSB (Least Significant Bit) first.

## Clock Modes: CPOL & CPHA

This is the most confusing part of SPI. Two config bits define timing:

- **CPOL** (Clock Polarity) — what state is the clock in when idle?
  - `CPOL=0` → clock idles LOW
  - `CPOL=1` → clock idles HIGH

- **CPHA** (Clock Phase) — which edge is data sampled on?
  - `CPHA=0` → sample on **leading** (first) edge
  - `CPHA=1` → sample on **trailing** (second) edge

Combined → 4 SPI Modes:

| Mode | CPOL | CPHA | Clock idle | Sample edge |
|---|---|---|---|---|
| **0** | 0 | 0 | LOW | Rising ↑ |
| **1** | 0 | 1 | LOW | Falling ↓ |
| **2** | 1 | 0 | HIGH | Falling ↓ |
| **3** | 1 | 1 | HIGH | Rising ↑ |

> **Mode 0 is by far the most common.** Most sensors, flash chips, and displays default to it. Using the wrong mode gives garbage data.

```
Mode 0 timing (CPOL=0, CPHA=0):

SCLK:  ___↑‾‾↓___↑‾‾↓___↑‾‾↓___
MOSI:  ═══[  B7  ][  B6  ][  B5  ]═
              ↑ sample   ↑ sample
```

## Multi-Slave Configurations

### Independent (Star) — most common

```
Master
  ├── CS0 ──► Slave 0
  ├── CS1 ──► Slave 1
  └── CS2 ──► Slave 2
       (SCLK, MOSI, MISO shared)
```

- Each slave has its own CS pin from the master
- Only one CS is asserted at a time
- Non-selected slaves must **tri-state** their MISO line

> Downside: more GPIO pins needed as slave count grows.

### Daisy-Chain

```
Master → [Slave 0 MOSI→MISO] → [Slave 1 MOSI→MISO] → back to Master MISO
```

- All share one CS line
- Data is shifted through all slaves in series
- Useful for shift registers, LED drivers (e.g. 74HC595)
- Latency grows with chain length

## SPI Variants (the xSPI Family)

Standard SPI sends 1 bit per clock on dedicated lines. Extensions add more data lines:

| Variant       | Data Lines        | Bits/Cycle | Notes                         |
| ------------- | ----------------- | ---------- | ----------------------------- |
| 3-Wire SPI    | 1 (bidirectional) | 1          | MOSI+MISO merged, half-duplex |
| Standard SPI  | 1+1 (MOSI+MISO)   | 1 each way | Full-duplex                   |
| Dual SPI      | 2 (IO0, IO1)      | 2          | Half-duplex data phase        |
| **Quad SPI**  | **4 (IO0–IO3)**   | **4**      | [[QSPI]]                      |
| **Octal SPI** | **8 (IO0–IO7)**   | **8**      | [[OSPI]]                      |

> In Dual/Quad/Octal: the CMD phase often still goes over 1 line (SPI mode), then the data phase switches to multi-line mode.

## SPI vs I2C vs UART

| Property | SPI | I2C | UART |
|---|---|---|---|
| Lines | 4+ | 2 | 2 |
| Speed | Up to 50+ MHz | Up to 5 MHz | Up to ~20 Mbps |
| Duplex | Full | Half | Full |
| Multi-slave | Yes (CS per slave) | Yes (addressing) | No (point-to-point) |
| Complexity | Medium | Low | Low |
| Protocol overhead | None | Address + ACK | Framing bits |
| Typical use | Flash, displays, ADC | Sensors, EEPROM | Debug, GPS, BT |

## Common Gotchas

- **Wrong SPI mode** → data is garbage; always match CPOL/CPHA to device datasheet
- **CS not deasserted between commands** → some flash chips require CS HIGH between operations to latch a command; missing this causes silent failures
- **MISO floating** → non-selected slaves must tri-state MISO; without external buffers on shared buses this can corrupt reads
- **Clock speed too high** → signal integrity degrades, especially on long traces or breadboards; start slow and increase
- **MSB vs LSB first** → most devices are MSB first, but some (e.g. certain SD cards in SPI mode) differ
- **Max clock frequency** → not a fixed number — it depends on the specific device *and* the operating voltage (lower voltage = lower max clock in some parts)

---

## Common Devices on SPI

- NOR Flash (W25Qxx, MX25Lxx)
- SD cards (in SPI mode)
- OLED/TFT displays (ILI9341, SSD1306)
- ADC/DAC chips (MCP3204, DAC8565)
- IMU sensors (MPU-9250, ICM-42688)
- RF transceivers (CC1101, nRF24L01)
- SRAM chips (23LC1024)

---

## Quick Summary

- SPI = clock + chip select + 2 data lines (MOSI, MISO)
- Full-duplex, synchronous, master-slave
- 4 clock modes from CPOL/CPHA combos — **Mode 0 is the default**
- No addressing — CS line selects the slave
- Fast and simple, but needs more pins than I2C
- Foundation of Dual SPI → QSPI → OSPI
